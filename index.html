01

RadioMapSeer Dataset

The primary dataset for this project. Each sample maps a city layout and transmitter position to a pathloss radio map. The reconstruction task: recover the full map from sparse signal measurements.

RECONSTRUCTION TASK RadioMapSeer • Urban pathloss
Input City map + Tx + sparse measurements
Output Full 256×256 pathloss map
Samples 56,080 maps
Resolution 1 meter / pixel
Cities 701 real layouts
Propagation Ray-tracing (DPM/IRT)
Pixel values Signal strength (dB)
Temporal None (static)
Key property: Each radio map is a static snapshot — computed once for a fixed environment via ray-tracing. There is no channel variation over time, no user mobility, and no Doppler effect. The reconstruction task has no natural deadline.

The first email describes the reconstruction task using Bilginer’s models. The follow-up emails introduce channel coherence time and Doppler spread — wireless channel concepts.

RadioMapSeer is an RF dataset, so the connection is clearer — but it is still static. Each map is computed once for a fixed environment, with no channel variation over time.

In the emails, “kanal” refers to a wireless communication channel that changes over time due to mobility. But which channel are we reconstructing? A channel that decorrelates in 3–200ms would require time-varying channel measurements (e.g., CSI matrices sampled as a user moves) — neither dataset provides this.

It is worth noting that the RadioMapSeer literature itself frames the task as an offline network-planning surrogate for ray tracing — not as a real-time system. No RadioMapSeer paper discusses edge deployment, on-device inference, or mobile-phone latency constraints. The implicit deployment target is a GPU-equipped server for planning workflows.

What I’d like to clarify: How do static field reconstruction and time-varying channel coherence connect? Which “channel” is being reconstructed, and what does the input/output look like in the coherence time scenario?

The professor’s example uses a 2-second inference time to motivate the problem — but the actual models run at 1–3 ms per sample, which is 1000× faster than that example. At these speeds, the router’s own overhead becomes the bottleneck.

Beyond latency: how does the router decide? If it routes based on coherence time (a known system parameter), no ML is needed — just a threshold. If it routes based on sample difficulty, the router only sees sparse observations. The literature does show that scene difficulty varies (building density, LOS/NLOS), so heterogeneity exists — but detecting it cheaply enough to justify the router overhead is an open question.

From what I can find, no published paper applies early exit, model routing, or input-adaptive depth to radio map prediction. The closest work is REM-U-net, which trains two models (low-density vs high-density scenes) and selects between them based on building density — but that is motivated by accuracy, not latency. This suggests the idea is novel but also untested.

What I’d like to clarify: What signal does the router use to make its decision? And given the ~2ms baseline, what is the target latency range where routing provides a meaningful improvement?
03

The Disconnect

The current motivation assumes a time-varying wireless channel with strict deadlines, but the benchmark tasks themselves are static per sample.

🚗

Vehicular

∼3 ms

At 100 km/h, the channel decorrelates in about 3 milliseconds. Inference must complete before then.

🚶

Pedestrian

∼50 ms

Walking speed gives more headroom, but the channel still evolves continuously.

💻

Stationary (indoor)

∼200 ms

No user movement, but surrounding objects move. Maximum coherence time — still finite.

RadioMapSeer  STATIC — Fixed environment, computed once. No channel variation over time. No natural inference deadline. Ray-tracing output, not live measurements.
The core mismatch: Coherence time creates genuine latency pressure only when the channel is changing over time. RadioMapSeer is a static per-sample benchmark, so it does not impose a natural coherence deadline. That does not mean speed is irrelevant, but it does mean the current real-time RF framing needs a stronger justification — either a different dataset or an explicit proxy framing.
Literature support: RadioMapSeer-style prediction is usually posed as a static snapshot task: environment map + single Tx → dense pathloss map. The dominant framing in that literature is fast surrogate modeling for propagation simulation / planning. Stronger real-time digital-twin claims appear in newer radio-map papers, but not cleanly in the core RadioMapSeer benchmark.
Questions for discussion:
  • Should we use a time-varying channel simulator (QuaDRiGa, DeepMIMO, Sionna) if coherence time is the real motivation?
  • Or is RadioMapSeer intended as a proxy task for adaptive spatial reconstruction, in which case we should replace the coherence-time story with a planning / repeated-query / simulation-surrogate story?
04

Router Feasibility

Even if we accept a speed motivation, adaptive inference has limited upside on the canonical task because the dominant CNN baselines are already in the low-millisecond regime.

Model Type Inference / map
RadioUNet CNN 1–10 ms
PMNet CNN “a few ms”
AIRMap CNN 4.2 ms
RadioMamba Mamba+UNet 28 ms
RadioDiff Diffusion 553 ms
Ray tracing (CPU) Simulation 86–387 s
Literature support: Directly reported forward times for RadioMapSeer-style CNN predictors are already roughly 1–10 ms (RadioUNet), about 10 ms in the ICASSP challenge overview, and 14 ms for a REM-U-net variant. The clearest latency pain in the literature appears instead in diffusion-based radio map generation, where inference rises to hundreds of milliseconds or seconds.

1 Router overhead

A router network adds its own latency. Even 0.2ms overhead is a 10% tax on a 1.9ms forward pass. When the baseline model already runs in a few milliseconds, the absolute headroom for routing is small unless the deployment loop is extremely throughput-constrained.

2 Difficulty detection from sparse input

The literature does show heterogeneous difficulty — LOS vs NLOS, building density, map inaccuracies, and missing buildings all affect error. So routing is not conceptually absurd. The weaker point is that this heterogeneity may not buy much when the baseline CNN forward pass is already only a few milliseconds.

3 Coherence time is already known

If routing is driven mainly by coherence time, then the system already knows the deadline from Doppler assumptions. In that case a learned router is hard to justify as the contribution; a threshold rule or fixed policy may explain most of the benefit.

Questions for discussion:
  • Are we targeting a setting where adaptive inference is actually valuable: diffusion-scale latency, repeated-query optimization, CPU / edge deployment, or heavy preprocessing?
  • If the target is still the canonical CNN task, would early exit inside one model be more defensible than routing between separate models?
  • What exact baseline are we trying to beat — always using the fast model, always using the accurate model, or replacing a much slower simulator?
05

Questions for Discussion

Concrete items to resolve before proceeding with implementation.

  1. Dataset choice

    Is RadioMapSeer the final experimental dataset, or should we consider time-varying channel simulators (QuaDRiGa, DeepMIMO) to match the coherence time story?

  2. Routing trigger

    What decides which model to use — the coherence time (a known system parameter requiring no ML) or the estimated difficulty of the reconstruction input (requires a learned router)?

  3. Routing vs. early exit

    Given very fast baseline inference (1–3ms), would early-exit within one model avoid the overhead problem of a separate router? This changes the architecture and the research contribution.

  4. Target scenario

    What is the deployment scenario — vehicular, pedestrian, indoor? This determines the coherence time range and the reconstruction requirements. It also determines whether the ~2ms inference of current models is already fast enough.

  5. CNN vs. diffusion routing as an alternative angle?

    The literature shows a large latency gap between CNN models (1–10 ms) and diffusion models (553+ ms), while diffusion models offer higher accuracy on hard scenes. Routing easy scenes to a CNN and hard scenes to a diffusion model could be a viable and novel direction — no paper has tried this. Though architectures like RadioMamba (28 ms) are closing this gap.

  6. Target venue & timeline

    What conference or journal are we targeting, and what is the submission deadline? This shapes the scope of experiments we can run.
